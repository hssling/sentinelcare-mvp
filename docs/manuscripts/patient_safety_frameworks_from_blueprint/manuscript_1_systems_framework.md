# Manuscript 1: SentinelCare Systems Framework for Computable Patient Safety

## Proposed Title

SentinelCare: A Layered Systems Framework for Computable, Real-Time, Human-Governed Patient Safety

## Abstract

### Background

Preventable harm is generated by failures distributed across people, process, communication, timing, and system design. Existing safety tools are frequently retrospective, siloed, and weakly connected to real-time intervention workflows.

### Objective

To define a systems-level patient safety framework that operationalizes risk detection, escalation, closure, and learning as computable functions across care settings.

### Framework

SentinelCare is specified as an eight-layer platform: Data Fabric, Clinical Knowledge, Detection, Prediction, Intervention, Learning, Governance, and Experience. The framework uses an explicit clinical process chain (Intent -> Action -> Confirmation -> Response -> Follow-up -> Closure) as the unit of safety analysis. Errors are formalized as omissions, contradictions, unsafe delays, or unresolved closure states.

### Implications

The framework reframes patient safety from incident counting to continuous process control. It provides a translational architecture for staged deployment with human-in-the-loop oversight, measurable operational metrics, and governance-ready auditability.

## 1. Introduction

Patient safety remains constrained by fragmented information, delayed recognition of risk, and absence of closed-loop accountability in routine workflows. Existing systems often identify isolated events without integrating:

1. cross-modal evidence,
2. role-specific responsibility,
3. escalation timing,
4. closure verification,
5. post-event learning.

The SentinelCare blueprint proposes a prospective safety layer that treats care as a dynamic process network rather than static documentation artifacts.

## 2. Conceptual Position

SentinelCare is not framed as autonomous clinical authority. It is a safety-control infrastructure with three principles:

1. human oversight at high-risk decision points,
2. precision-oriented alerts over high-volume interruption,
3. auditability and override capture as mandatory design elements.

## 3. Core Process Model

The framework encodes each clinical pathway as:

Intent -> Action -> Confirmation -> Response -> Follow-up -> Closure

This chain is the analytic primitive. A safety event is declared when one or more states occur:

1. missing expected action,
2. contradictory evidence,
3. harmful delay,
4. unresolved ownership,
5. closure failure.

## 4. Layered Architecture

### Layer 1: Data Fabric

Purpose:

1. source connectivity (HL7/FHIR/DICOM),
2. normalization and identity/time resolution,
3. event-quality harmonization.

### Layer 2: Clinical Knowledge

Purpose:

1. codify SOP/guideline/formulary logic,
2. encode escalation policies and checklist rules,
3. maintain site-governed versioning.

### Layer 3: Detection

Purpose:

1. deterministic rule execution,
2. process deviation identification,
3. contradiction detection and extraction across modalities.

### Layer 4: Prediction

Purpose:

1. anticipatory risk scoring (deterioration, ADE, follow-up risk),
2. horizon-based forecasting for pre-harm intervention windows.

### Layer 5: Intervention

Purpose:

1. role-aware signaling (silent, nudge, interruptive),
2. escalation workflows,
3. corrective-action launchers.

### Layer 6: Learning

Purpose:

1. override and feedback analysis,
2. false-positive/false-negative review loops,
3. drift and adaptation signals.

### Layer 7: Governance

Purpose:

1. policy and model lifecycle control,
2. approval and rollback pathways,
3. explainability and audit stores.

### Layer 8: Experience

Purpose:

1. role-native worklists and command views,
2. operational transparency across input-analysis-output.

## 5. Multidomain Applicability

The framework supports ten safety domains in the blueprint, with MVP prioritization of:

1. medication safety,
2. critical-result closure safety,
3. deterioration/sepsis surveillance.

This preserves architectural breadth while constraining deployment scope for measurable implementation.

## 6. Operational Design Logic

The framework imposes six operational invariants:

1. every alert must state evidence, owner, deadline, and action,
2. every high-severity signal must support explicit escalation path,
3. every intervention must permit accountable override capture,
4. every pathway must support closure verification,
5. every release must remain policy-governed,
6. every production cycle must generate validation artifacts.

## 7. Evaluation Framework

Primary metrics are process-outcome linked:

1. alert PPV/sensitivity,
2. time-to-detection,
3. time-to-acknowledgment,
4. time-to-action,
5. closure completeness,
6. override and no-action-after-alert rates.

## 8. Discussion

The SentinelCare systems framework contributes a coherent bridge between technical architecture and clinical safety operations. Its value lies in integration: the same framework that detects risk also carries responsibility, escalation, closure, and governance state.

The immediate next requirement is prospective pilot validation under controlled governance with narrowly scoped alert classes.

## 9. Conclusion

SentinelCare defines a computable patient safety architecture that is prospective, modular, human-governed, and measurable. It establishes a robust theoretical base for staged translation from prototype to validated clinical operations.
